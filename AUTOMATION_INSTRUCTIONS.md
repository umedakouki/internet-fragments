# 日次インターネット・コレクション指示書

## 目的

毎日ネットサーフィンを行い、まだ一般的な収集ジャンルとして輪郭が固まっていないが、インターネット上に多数存在し、並べることで文化的・視覚的・歴史的価値が立ち上がる断片を保存する。

## 毎日の手順

1. `git status --short --branch` を確認し、既存の未コミット変更を把握する。自分の作業と無関係な変更は戻さない。
2. 自動化メモ、`diary/`、`data/collections/`、公開サイトを読み、既存ジャンル、前回の失敗、次回候補を把握する。
3. 検索語を固定しすぎず、ニュース、画像アーカイブ、デジタル資料館、個人サイト、過去のウェブ文化などを短時間ネットサーフィンする。
4. 当日のジャンルを1つ決める。既存ジャンルの継続、分離、結合、改名も許可する。
5. そのジャンルから10～50件を保存する。各項目に最低限、タイトル、原典URL、取得日、短い選定理由を持たせる。作者・制作日・ライセンスが判明する場合は必ず保存する。
6. 写真や資料は、再利用可能なライセンス、パブリックドメイン、または引用・紹介に必要な範囲を優先する。権利条件が不明な場合はファイル本体を複製せず、メタデータと原典リンクのみを保存する。
7. 重複、リンク切れ、出典不明、機械生成スパムを点検する。個人情報、違法性のある資料、露骨な搾取につながる資料は収集しない。
8. `diary/YYYY-MM-DD.md` に、探索経路、選定理由、結果、失敗と修復、トレンド所感、次回候補を書く。
9. `data/collections/YYYY-MM-DD.json` を作成し、`data/collections/index.json` に日付、題名、件数、JSONパス、日記パスを追記する。索引は日付昇順を保つ。
10. 再利用可能な画像は `assets/collections/YYYY-MM-DD/` に保存する。再配布条件が不明な資料は画像本体を保存せず、メタデータと原典リンクだけをJSONに記録する。
11. 共通検証スクリプトを実行し、件数、索引、パス、画像、重複、権利情報を確認する。
12. ローカルHTTPサーバーと実ブラウザーで、PC・モバイル表示、最新表示、過去コレクション選択、`?collection=YYYY-MM-DD`、絞り込み、詳細表示、原典リンク、コンソールエラーを確認する。
13. 変更対象だけをGitでコミットし、`main` へプッシュする。GitHub Pagesのデプロイ完了を待ち、HTML、索引JSON、日付別JSON、代表画像へ公開URLから到達できることを確認する。

## 実行上の標準

### UTF-8とPowerShell

- 日本語を含むファイルは必ず `Get-Content -Encoding UTF8` で読む。文字化けした内容を推測して処理しない。
- Windows PowerShell 5でUTF-8 BOMなしの収集スクリプトを直接実行すると、日本語リテラルが壊れる場合がある。次のラッパーを使う。

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/run-utf8.ps1 scripts/collect-YYYY-MM-DD.ps1
```

### 取得とレート制限

- APIメタデータは可能な限り一括取得し、画像取得は直列で行う。項目ごとのAPI呼び出しを避ける。
- 画像取得済みファイルは再実行時に再取得しない。途中失敗から再開できるスクリプトにする。
- HTTP 429や一時的な5xxでは、待機時間を増やしながら再試行する。短時間の連続再試行は行わない。
- 別配布元へ切り替える場合は、ファイルID、作者、撮影内容などで同一資料と確認できる原配布元だけを使う。メタデータの基準URLと画像取得元の関係を日記に残す。
- TLS検証の無効化は常用しない。環境固有の証明書問題で必要になった場合だけ使い、理由を日記に残す。

### 機械検証

日付別JSONと索引を作成した後、次を実行する。

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/validate-collection.ps1 -Date YYYY-MM-DD
```

公開前に原典リンクも確認する場合は `-CheckLinks` を追加する。リンク確認は先方への負荷を避けるため直列で行う。

### 実ブラウザー検証

- 内蔵ブラウザーを優先する。利用不能なら、インストール済みChromeまたはEdgeのヘッドレス実行を使ってよい。
- ローカルHTTPサーバーはブラウザー検証と同じ工程内で起動し、HTTP応答を確認してからブラウザーを開き、終了時に停止する。前工程で起動したサーバーが残っていると仮定しない。
- スクリーンショットだけで完了とせず、日付選択、フィルター、詳細ダイアログ、原典リンクを実際に操作する。
- モバイルは実幅390px前後で確認し、`document.documentElement.scrollWidth <= document.documentElement.clientWidth` も検査する。Windows版Chromeの最小ウィンドウ幅による切り抜きをモバイル表示と誤認しない。
- 検証用スクリーンショットやブラウザープロファイルは `output/` に置き、コミットしない。

### GitHub Pages公開確認

- `gh` がPATHにない場合は `.tools/bin/gh.exe` を使う。
- プッシュ後は `gh run list` と `gh run watch <run-id> --exit-status` でPagesデプロイの成功を確認する。
- CDNキャッシュを避けるため、確認URLには `?v=<commit>` を付けてよい。
- 少なくとも公開HTML、`data/collections/index.json`、日付別JSON、代表画像1件がHTTP 200で取得できることを確認し、公開索引と日付別JSONの件数も照合する。

## 継続・凍結・再編

- 10件を安定して見つけられない、出典や権利を追えない、類似物ばかりで差異が立ち上がらない場合、そのジャンルを `frozen` として日記に記録し、次回以降の収集を凍結する。
- 新ジャンルを思いついた場合は遠慮なく開始する。既存ジャンルの境界が不自然なら、結合・分離・階層化・改名を行い、変更理由を日記とサイトに残す。
- 現在のトレンドだけでなく、検索トレンド、過去のニュース、ウェブアーカイブ、廃れたサービス、歴史資料にも目を配る。
- 収集件数の多さより、項目間の比較可能性と出典の追跡可能性を優先する。

## 完了条件

- 1ジャンルにつき10～50件が保存されている。
- 日記が更新されている。
- サイトに新しいコレクションが表示され、原典と権利情報へ到達できる。
- 公開先への更新が成功している。失敗した場合は日記に原因と再試行方法を残す。
- `data/collections/index.json` と日付別JSONの件数・パスが一致している。
- 公開サイトの最新表示と日付指定表示が正常に動作している。
- 作業ツリーが意図した変更だけで構成され、検証用の `output/` がコミット対象に含まれていない。
