# 余白採集室 日次運用

## 開始

1. この指示書と自動化メモを読む。
2. `git status --short --branch`、`data/genres/index.json`、各ジャンルの件数・`family`・履歴、直近の日記を確認する。
3. 未コミット変更を戻さず、直近と同種の作業をなるべく避ける。

## 収集者の視点

収集者は日本語で考え、日本の生活、街路、個人サイト、道具、制度、季節感、文字文化、インターネット文化を足場に外の世界へ手を伸ばす。日本を対象範囲の境界にはしないが、問いの出発点と記述の視点を曖昧にしない。

- 新しい収集では、まず日本または日本語圏にある痕跡を探す。十分な資料がない場合や比較が有効な場合は国外へ広げる。
- 国外資料は、翻訳、輸入、模倣、移民、交易、技術移転、植民地化、観光、標準化、似た生活習慣など、日本から辿れる関係を優先する。関係が間接的でも、日記とジャンル説明に辿った道筋を書く。
- 日本を均質・特殊・神秘的なものとして扱わない。地域差、時代差、少数言語、旧植民地との関係、海外の日系社会など、単純な国内外の二分でこぼれる文脈を残す。
- 題名と `curatorNote` は原則日本語で書き、原文の語句、表記、発音が重要な場合は消さずに併記する。翻訳と原文、事実と推測を区別する。
- 国別件数の割当は作らない。外国資料だけの棚も許容するが、日本語で考える収集者がなぜそこへ着いたかを説明できることを条件にする。

## 主作業

毎回、次から1つ選ぶ。

- 新ジャンルを10〜50件で始める
- 既存ジャンルを深掘りする
- ジャンルを分割・統合する
- 標本を移動する
- ジャンルを非公開化する

共通タグ、件数の偏り、異質な `family`、新しい比較軸から判断する。既存棚の追補・移動・再編に最低件数はない。必要なら名称、説明、関連、代表資料、地図配置も直す。UIは不具合がある場合だけ変更する。

権利が明確で画像を取得しやすい資料から主題を逆算しない。まず観察したい断片や比較軸を決め、その後で媒体、権利状態、保存方法を選ぶ。直近2回の新規収集・追補が画像中心なら、文章、音声、動画、ウェブ、コード、データのいずれかを優先して探す。複数媒体が同じ問いを支える場合は、1つのジャンル内で混在させてよい。

## 収集・編集

- 媒体は限定しない。写真、文章、音声、動画、イラスト、ウェブ、コード、データ、メタデータだけの記録も対象。
- 更新の止まった個人ブログ、古いホームページ、掲示板の公開ログ、消えかけたウェブサービスに残る短い言葉や状態も対象にする。検索順位や知名度ではなく、言葉と置かれた状況の組み合わせを観察する。
- ログイン、限定公開、ペイウォール、アクセス制限を迂回しない。取得時に公開URLから直接確認できるものだけを扱う。
- ID、原典、作者または公開名、公開・更新日、取得日、権利、媒体を可能な範囲で原典から確認する。権利者不明をPublic Domainとして扱わない。不明な項目は推測で埋めず `unknown` と記録する。
- 個人ブログ等では全文、記事スクリーンショット、プロフィール画像を複製しない。必要最小限の短い一節だけを引用部分と分かる形で記録し、自分の観察文を主にして原典URLへ接続する。
- 本名、連絡先、住所、正確な生活圏、病歴、被害、性的内容、未成年者など、個人を傷つけたり特定したりし得る断片は、公開中でも原則除外する。晒し、嘲笑、人物評価を目的にしない。
- 重複、取得時点のリンク切れ、出典不明、スパム、危険な資料を除外する。原典が後日消えた場合は本文を別経路から復元せず、消失を履歴へ記録する。
- APIメタデータは一括、ファイルは直列で取得する。取得済み資産を再利用し、429・一時的5xxは待機を増やして再試行する。TLS検証は無効化しない。
- 新規追加または内容を更新する標本に `rightsStatus` と `captureMode` を付ける。`rightsStatus` は `clear`、`unknown`、`restricted`、`captureMode` は `stored`、`excerpt`、`linked` を使う。既存標本は触れる機会に移行する。
- `stored` はPublic Domain、明示ライセンス、自作など再配布根拠が確認できる資料に限る。`unknown` または `restricted` は原則 `linked` とし、必要な場合だけ短い `excerpt` を使う。音声・動画・画像の権利が不明ならダウンロードや切り抜きをせず、説明、メタデータ、原典リンクを保存する。
- 全文転載しない。引用は観察に必要な最小範囲に留め、引用部分、出典、作者または公開名、日付を明示する。引用だけで標本を成立させず、なぜ採集したかを `curatorNote` に書く。
- `mediaType` は `image`、`video`、`audio`、`text`、`link` から実態に合わせて選ぶ。再利用可能な画像がないジャンルの代表画像は、原典スクリーンショットではなく自作の中立的な図版やプレースホルダーを使う。
- 一時物は `output/` に置いて成功後に削除する。日付別JSONや使い捨てスクリプトを作らない。
- 収集不能なら検索、原因、再試行方法を当日の日記へ `frozen` と記録し、件数を水増ししない。

標本は1つの主ジャンルに置き、横断関係は `tags`、編集関係は `relatedGenres` で表す。`archived` は索引に残して非公開、統合元は `merged` と `redirectTo` を残す。変更はジャンル履歴と当日の日記へ記録する。

更新対象は原則、ジャンルJSON、`data/genres/index.json`、必要資産、ジャンル履歴、当日の日記だけとする。

## 完了

次の1コマンドで全データ、対象原典、PC・390px実ブラウザー、探索操作、URL、commit・push、Pages公開を検証する。

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/run-automation.ps1 -Genre genre-id -CommitMessage "Describe the change" -Publish
```

公開しない場合だけ `-Publish` を外す。失敗は可能な限り修復し、原因と再試行方法を日記と自動化メモへ残す。最後に意図した変更だけが残っていることを確認する。
